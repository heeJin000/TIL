# I. 개발 시스템 구조
- *사용자가 사용하는 프로그램은 웹과 앱으로 나누어진다. 이들은 세 가지 구조로 이루어진다.*  
## 웹
### 백엔드 / 프론트엔드 / 데이터베이스
+ 정보를 표시하는 Front-End, 정보를 핸들링하는(다루는) Back-End, 정보를 저장하는 DataBase 
+ 디비에 데이터를 저장하거나 조회하는 등 처리하면서  
 논리적으로 필요햔 정보를 핸들링하는 형태의 개발을 하는 것이 백엔드 개발자 (DB+Back-End)
+ 백엔드 개발자가 건네준 정보를 화면에 뿌려주는 담당이 프론트앤드 개발자 (Front-End)

### 비즈니스 로직 
(요즘에는 잘 안 쓴다고 함)
- 백엔드 단에서 여러 업무 처리를 위해 업무적인 요소를 붙여 프로그램을 개발하는 것  
  ➡️ 그냥 프로그램 개발하는 거구나  
<br>

## 앱
### Android-IOS / 백엔드 / 데이터베이스
- Android-IOS는 프론트 엔드와 같이 정보를 화면에 표시한다. 대신 Front-End 개발자가 아닌 앱개발자라고 부른다.
- 백엔드와 데이터베이스는 웹과 기능이 같다.  
➡️ 앱을 만든다고 가정하면 앱 개발자 뿐만 아니라 백엔드 개발자도 필수로 있어야 한다.
<br><br>

## 논리적 구조 구분

### 웹 논리적 구조 구분
- SSR(서버사이드랜더링)
  + 서버개발 하는데 Front-End, Back-End 구분하지 않는 케이스.  
  + 과거에 많이 썼고 웹만 필요할 때 사용하던 방법
  + Front-End, Back-End ↔️ DB

- CSR(클라이언트사이드랜더링)
  + Front-End, Back-End 명확히 구분.
  + 백엔드와 프론트 엔드 사이 API
  + Front-End ↔️ API, Back-End ↔️ DB


|SSR(서버사이드랜더링)|CSR(클라이언트사이드랜더링)|
|---|---|
|프, 백 구분하지 않는다|프, 백 구분한다|
|웹개발만|웹, 앱개발을 같이|
|과거에 많이 사용|현재에 많이 사용|
<br>

### 앱 논리적 구조 구분
Android & iOS ↔️ Back-End, API ↔️ DB
<br><br>

### 웹 + 앱 논리적 구조
- 앱과 웹의 논리적 구조를 합친 모습.
- 하나의 백엔드 개발로 리소스 낭비를 줄이고 화면을 각각 Front-End (웹) / Android, iOS (앱)으로 개발.
<br>

### 서버 / 앱 개발 구분
- 서버 개발
  + 만든 프로그램을 서버에 배포
  + Front-End, API, Back-End  

- 앱 개발
  + 만든 프로그램을 앱에 배포
  + Android, iOS

# II. 네트워크 기본 구조
## 네트워크 구조 용어
- 방화벽: 방화벽 또는 파이어월은 미리 정의된 보안 규칙에 기반한, 들어오고 나가는 네트워크 트래픽을 모니터링하고제어하는 네트워크 보안 시스템(위키백과)
- DMZ: 컴퓨터 보안의 비무장지대는 조직의 내부 네트워크와 (일반적으로 인터넷인) 외부 네트워크 사이에 위치한 서브넷
- 내부망: 물리적 망분리, 접근통제시스템 등에 의해 인터넷 구간에서의 접근이 통제 또는 차단되는 구간  
<br>

## 네트워크 구조
(방화벽 밖)해킹과 같은 나쁜 짓, 위험과 안전이 뒤섞인 인터넷
- ------방화벽-------
  + DMZ(최소 안전 영역)
  + (약한 정책: 정보 소통 많이 필요)  
  + (안전한 인터넷 접근 허용)
  + (위험한 인터넷 배척)
  + 외부와 소통 업무 지역
  + 개발/테스트 서버와 개발 DB가 여기 있음
    - ------방화벽-------
      + 내부망(절대 방어 영역)
      + (강한 정책: 꼭 필요한 정보만)
      + (안전한 인터넷 접근 허용)
      + (위험한 인터넷 배척)
      + 소통도 여기 안에서만
      + 고객의 돈, 개인 정보 등의 운영 서버와 운영 DB가 여기 있음

# III. 네트워크 이중화
## 서버 최소 구성
네트워크는 기본적으로 DB Server에 필요한 데이터를 저장하고 WAS를 돌리면 작동  
- WAS(고유 IP 숫자는 의미 파악이 어려움. -> 고유이름 도메인으로): 웹 프로그램이 돌아가는 서버  
👿 망가지면 서비스 불가  
↕️
- DB Server:  데이터를 저장 관리하는 서버  
👿 망가지면 데이터 소실

## WAS 서버 이중화 구성
- 이중화 :  하나 고장나면 큰일나니 2개를 준비해두는 것. 하나 고장나면 나머지 하나가 버티는 새 문제 해결하기
- 하나의 DB Server를 공유하는 WAS 2개. maver1 과 maver2.  
🖥 maver1이 고장났으니 maver2로 접속을 유도해야 하는 상황.
- L4(고유 IP 숫자는 의미 파악이 어려움 -> 고유이름 도메인으로): WAS 서버 주소지를 찾아가게 하는 정보가 되는 장비  
  ex) maver 접속할게요 -> maver1이 고장났네? maver2로 가세요.  
      maver1 지금 너무 바빠요. -> 그래? 이건 maver2에서 처리해야겠다.  
      (로드밸런싱: 부하(로드)가 걸리지 않게 균형있게 일처리하는 것)
  + 로드 밸런싱 방법
  1. Round Robin:  순차적으로 처리(1 ➡️ 2,  2 ➡️ 1)
  2. Least Connection(커넥션: 데이터 주고 받는 통로):  
     maver1에 1분 걸리는 조회를 동시에 10 사람이 접근 => 총 10분 (통로가 너무 바빠)
     maver2에 1초 걸리는 조회 동시에 3 사람이 접근 => 총 3초 (통로 여유로움)
     => maver2에다 일 물어다 줌
  3. Fatest Response Time:  신호를 보냈을 때 피드백 빠른 서버에게 일 줌
  4. Source Hash Scheduling:  접근하는 pc IP에 따라서 서버 배정  

  *(L4 이름? OSI 7 계층의 4번째 계층의 업무 처리를 하는 장비)*

 # IV. 네트워크 연결
 ## IP 사전 지식 
 - IP :  (Internet Protocol) 각 장치를 나타내는 IP 주소
 - 장치:  서버, 컴퓨터, 네트워크 장비
 - 주소:  상대가 나를, 내가 상대를 찾기 위해 필요한 주소  

 - IP는 ID의 역할을 한다고 봐도 됨
 - 다른 장치에 접속하려 할 때, 그 쪽에서 나를 들여보내 줄지 ID 확인

## Public IP VS Privae IP
- public IP: 외부에서 나를 찾는 주소(공인 IP, 외부 IP)
- Private IP: 내부에서 나를 찾는 주소(사설 IP, 내부 IP)  
  *예시) public IP는 집주소 (우편 배달 가능)*         
        *Private IP는 집 안 방 주소(우편 배달x)*  
### 내 PC에서 Public IP와 Private IP 구분법
1. public IP 
네이버에 '내 IP' 라고 검색 (내부 IP가 아니라 우리집으로 연결된 원선에 대한 IP 주소가 검색됨)

2. private IP
cmd에서 'ipconfig' 라고 입력  
내부적으로 접근 가능한 내부 주소지 찾을 수 있음
<br><br>

## 전용선과 VPN 필요 이유
- 집에서 회사 서비스 들어가려고 하니까 내가 들어온 경로를 따라 누가 정보를 납치하거나 파괴하는 악용 사례가 벌어질수 있음
- 해결책: 물리적으로 밖에 있는 것을 내부에 있는 것처럼 꾸미는 방법으로 해결
    
## 전용선 VS VPN
- 전용선(=직통선) 
  + 서버와 외부에 다른 서버를 직접 연결하는 물리적인 선. 정보의 안전이 보장되어 있음.
  + 장비를 두고 연결해야 하기 때문에 비용이 들고 관리가 힘듦.

 - VPN
  + 전용선의 장점을 해결하기 위해 발명
  + 밖으로 정보가 뺏기거나 이상한 게 안으로 들어와도 문제 처리를 핸들링 할 수 있다면...? 에서 출발
  + 가상의 전용선
  + 회사 외부망 앞단과 실제 인터넷을 사용하길 원하는 쪽에(지점 등) 물리적인 장비를 놓고 그것을 통해 인터넷을 이용  
    ➡️ 1. 출입 ID 인증(회사 외부망 통과할 때 인증, 지점망에 도착하면 인증 정보 있는지 확인하고 접근 허용)  2. 정보를 암호화(누가 정보를 빼가도 암호 처리된 상태)
  + VPN 장비가 있고 장비 대신 VPN S/W로 작업하는 경우도 있음

# V. 트랜잭션
## 트랜잭션 사전적 정의
- 데이터 베이스 상태를 변환시키는 하나의 논리적 기능을 수행하기 위한 작업의 단위
- 속성: 원자성, 일관성, 독립성, 지속성
- 트랜잭션 각각의 정의
  + All or Nothing 하든가 말든가. 중간은 없다.
  + DB: 중간에 멈추면 큰 일 나, 원래대로 돌려
  + 프로그램: 여기까지는 한번에 처리해야 해
  + 일반 업무: 최소 업무 단위

### All or Nothing
- ATM 기기: 천원 인출 하고 싶음 ➡️ 잔고 확인 ➡️ 잔고 있음 ➡️ 잔고 차감 ➡️ 천원 인출 
- 잔고 차감 하다가 문제가 생긴다면? ➡️ 천원은 못 받았는데 잔고는 줄어있는 문제 발생  

❗️ 돈을 지급할 때는 본사 계좌 차감 후 정상적으로 인출되는 것까지 확인하고 처리 확정(Commit) / 
그게 아니면 처음으로 되돌려(Rollback)  
<br><br>

## DB-개발 관계자들은...
- 개발 트랜잭션: 프로그램 전체 프로세스 (주문 - 결제 - 처리)
- DB 트랜잭션: 결제 금액 처리 - 주문 등록 처리  
<br>

## 일반 IT관계자들은...
- 일반 업무 = 최소 업무 단위
  + "이거 트랜잭션 처리 된건가요?" = 이 업무들 한번에 처리 되는 거냐
  + "트랜잭션 범위를 어디까지 해야 해요?" = 어디서부터 어디까지 (한번에 처리 / 문제 발생하면 원복) 해야 해요?
  + "하루에 트랜잭션이 몇 건이나 발생해요?" = 하루에 몇 건이나 구매 처리가 되나?

## 학사 기록 정보 수기 관리
- 학사 정보 기록 [창고]: 정보를 모아두는 공간
- 학생 기본 정보 [책상] / 학생 성적 정보 [책상]:  구체적인 정보가 올라오는 곳
- 실제 정보 [기록지]   / 실제 기본 정보 [종이]:   각각의 실제 정보가 기록되는 곳
  ➡️ 학생의 기본 정보가 필요하다면 학사 정보 기록 창고로 가서 학생 기본 정보 책상 위의  
    기록지를 보면 된다.
- 창고 = space, 책상 = table, 기록지 = record

### 정보를 찾을 때
*300 명 중에 학번 0150을 찾고 싶을 때?*
- 생각 없이 쌓으면 오래 걸리고 힘듦
- 학번 순으로 쌓으면 중간부터 보면 됨 (빠름 쉬움)
- 쌓아올리기 편한 속성 하나 정하기 ➡️ 학번
  + 학번 = "key"
  + 쌓아올리는 기준, 중복되지 않는 기준을 명확히 충족시킬 것 (Primary Key)

#### 인덱스
*이름이 홍길철인 학생을 찾는다면?*
- 학번으로는 홍길철을 빠르게 찾을 수 없음
- 이름을 하나의 주소록처럼 정리하자 ➡️ 이름, Table에서 어디 위치인지
- 새로운 데이터가 들어왔을 때 Table에 학번 순으로 삽입하고 인덱스에는 이름 순으로 정리 & 위치 표시
- 잘 쓰면 빠르고 효과적 / 못 쓰면 인덱스 찾는데 시간 더 걸릴 수도

# VII. 데이터베이스 응용
## DB를 쓰는 이유
- 정보를 저장해두고 (Create, Update, Delete)
- 필요에 따라서 꺼내어 봄 (Read)
➡️ CRUD
- DB에 CRUD를 하려면 뭔가 액션이 필요

## SQL
- SQL: Structured Query Language
  + DB에게 말하는 언어
  + 정해진 틀 안에서 질문처럼 
  <br><br>

### SQL 예제
*컴퓨터 공학과 학생들 정보만 보고 싶을 때*  
➡️ SELECT* FROM studentTable WHERE major='컴퓨터 공학과';
- 자연어(Natural Language): select, give, send 전부 가능. 순서도 상관 없음. 말만 되면 됨.
- 구조어(Structured Language): select만 가능. 정해진 순서로.


## SQL 기본 용어
- C: Insert 데이터를 넣고
- R: Select 데이터를 조회하고 
- U: Update 데이터를 수정하고
- D: Delete 데이터를 지우고
- R은 데이터의 변화가 없지만 나머지는 데이터의 변화가 있음
- 변화를 줄 때 실수하지 않게 ROLLBACK(원래대로 돌려), COMMIT(변경 확정)이 있음

## 표현 해석
- "쿼리를 날리다": DB에 데이터 CRUD 명령을 보내다
- "조회 쿼리가 느려서": 조회 명령 후 데이터 나오는 시간이 오래 걸린다
- "커밋 했니?": CUD 하고 확정했어?
- "데이터 다 롤백했어": 뭔가 잘못돼서(백업한 것 엉망됐다거나) OR 명령했던 것 전부 원복했다
